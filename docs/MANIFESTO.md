# DUS Manifesto

## The Old Assumption

Most interface systems still assume this order:

1. define boxes
2. stack boxes with rules
3. bolt on interaction
4. bolt on animation
5. bolt on semantics
6. bolt on uncertainty if AI forces you to

That order made sense when interfaces were mostly:

- forms
- pages
- cards
- menus
- predictable app chrome

It breaks down when the interface itself must express:

- evidence
- contradiction
- uncertainty
- relation strength
- focus
- reorganization under interaction

## The New Assumption

DUS starts from a different premise:

> an interface can be modeled as semantic nodes under targets, relations, losses, and hard constraints, and only then rendered onto one or more surfaces.

This changes the center of gravity:

- from box trees to semantic graphs
- from fixed rules to guided optimization
- from discrete view states to continuous reconfiguration
- from style-first UI to runtime-first UI

## The Actual Bet

The bet is not that every webpage should become fluid.

The bet is that some important interfaces, especially AI-native ones, are currently trapped inside the wrong substrate.

When content is probabilistic, relational, and revisable, box layout becomes a local compromise instead of a natural fit.

DUS tries to supply a better substrate.

## What Makes This Different

The novelty is not just GPU rendering.

The novelty is the unification of:

- layout
- interaction
- semantic relation
- confidence
- explainability

inside one runtime model.

In the strongest version of this idea:

- the solver decides where things should live
- the renderer decides how they should appear
- the host bridge decides how humans edit, select, or annotate them

These concerns stop being welded together inside a box-model stack.

## Why AI Changes the Stakes

AI systems do not emit certainty. They emit weighted, revisable structure.

Interfaces built on top of them need to express:

- confidence gradients
- contested claims
- linked evidence
- retrieval traces
- reorganizing attention

Traditional UI systems can fake this, but only by layering special-case logic on top of primitives that were never designed for it.

DUS treats those properties as first-class.

## What Success Would Look Like

Success is not “people say the shader looks cool.”

Success is:

- one real workflow becomes easier in DUS than in a card/list UI
- the runtime can explain why nodes moved where they moved
- the same solved layout can power both experimental and practical surfaces
- teams can build on the runtime without inheriting an opaque pile of magic

## What We Refuse

- pretending the fluid effect is the product
- pretending this already replaces CSS or React
- pretending optimization without explainability is acceptable
- pretending every interface wants continuous fields

## The Real Opportunity

If DUS works, it does not merely add a new visual style.

It offers a new interface substrate:

- semantic
- continuous
- inspectable
- multi-surface
- AI-native

That is a bigger ambition than a renderer and a narrower ambition than replacing all frontend development.

That narrowness is strength, not weakness.
